# План улучшения Clayfolk — 10.07.2026

## Цель

Превратить работающий набор систем в законченную короткую партию:

```text
развитие поселения -> предупреждение -> угроза -> защита -> восстановление -> финал
```

Главный приоритет — темп, угроза и понятный конец партии. Новые ресурсы, роли и крупные переделки откладываются до доказанного игрового ядра.

## Текущее состояние

Уже работают:

- добыча и перенос еды, глины, дерева и камня;
- зоны добычи и запрета;
- приоритеты работ;
- мастерская, топоры и кирки;
- костёр, склады, хижины, стены, ворота и идол;
- дождь, повреждение и починка стен;
- паук, бой и паника;
- победные цели и новая партия;
- авторитетная серверная симуляция и сохранение мира.

Проверено 10.07.2026:

- `npm run check` — PASS;
- `npm run check:slice` — 20/20 PASS;
- server benchmark: 160 жителей — 4.55 ms/step; 1000 жителей — 36.05 ms/step;
- размер сетевого снимка: 228–527 KB при 160 жителях, 849–1456 KB при 1000 жителях.

Главный долг:

- пока нет цельной напряжённой партии;
- `client/src/main.ts` вырос до 2122 строк;
- `server/src/sim/world.ts` вырос до 1112 строк;
- сценарная проверка крупная и недетерминированная из-за `Math.random()`;
- команды клиента объявлены только на сервере, клиент отправляет сырые JSON-объекты;
- сохранение не атомарное;
- документация местами отстаёт от кода.

## Этап 2. Фаза угроз

Главное следующее игровое ядро.

### 2.1 Паук приходит набегами

Файлы:

- `server/src/sim/enemy.ts`;
- `server/src/sim/step.ts`;
- `server/src/config.ts`;
- `shared/types.ts` при необходимости нового состояния;
- клиентский HUD в `client/src/main.ts`.

Задачи:

- [ ] У паука появились состояния ожидания, предупреждения, набега и отступления.
- [ ] Между набегами поселение получает спокойное время.
- [ ] Предупреждение видно игроку заранее.
- [ ] Сила и частота набегов задаются в `CONFIG`.
- [ ] Состояние набега переживает сохранение и загрузку.
- [ ] Headless-сценарий воспроизводит полный цикл набега.

### 2.2 Паук ломает оборону

- [ ] При контакте со стеной или воротами паук вызывает существующую механику `damageBuilding`.
- [ ] Паук выбирает доступную цель и не зависает навсегда у одной клетки.
- [ ] Повреждение хорошо читается визуально.
- [ ] После набега строители чинят повреждения текущей системой глины.

### 2.3 Тревога

- [ ] Добавить команду и кнопку «Тревога».
- [ ] Обычные жители прекращают дальние работы и возвращаются к поселению.
- [ ] Стража собирается возле ворот или ближайшей угрозы.
- [ ] Снятие тревоги возвращает обычное распределение работ.
- [ ] Тревога принадлежит поселению, а не всему миру.

### 2.4 Баланс угрозы

- [ ] Поселение без стен несёт заметные потери.
- [ ] Подготовленное поселение со стенами и стражей обычно выживает.
- [ ] После набега есть время на лечение, починку и пополнение запасов.
- [ ] Защита полезна, но не превращает набег в полностью пассивное ожидание.

Критерий готовности: предупреждение создаёт напряжение; тревога меняет поведение жителей; стены реально спасают; после атаки возникает фаза восстановления.

## Этап 4. Инженерная страховка

### 4.1 Детерминированная симуляция

- [ ] Заменить прямой `Math.random()` в симуляции на RNG мира с seed.
- [ ] Хранить seed и состояние RNG в durable snapshot.
- [ ] Печатать seed при падении сценарной проверки.
- [ ] Проверить одинаковый результат двух прогонов с одинаковым seed.

### 4.2 Разделить сценарные проверки

Текущий `server/src/bench/sliceCheck.ts` сохранить как общий smoke-test. Добавить отдельные сценарии:

- [ ] добыча и инструменты;
- [ ] строительство и ворота;
- [ ] дождь и починка;
- [ ] набег и тревога;
- [ ] победа и поражение;
- [ ] snapshot round-trip.

Каждый серверный шаг обязан проходить через `npm run check` и релевантные сценарии.

### 4.3 Надёжное сохранение

- [ ] Писать snapshot во временный файл.
- [ ] После успешной записи атомарно заменять основной файл.
- [ ] Обрабатывать `SIGTERM` вместе с `SIGINT`.
- [ ] Проверить загрузку повреждённого, старого и более нового snapshot.
- [ ] Не терять важное runtime-состояние мастерской, угроз и таймлайна.

Критерий готовности: баг симуляции воспроизводится по seed; оборванная запись не уничтожает последнее рабочее сохранение.

## Не делать сейчас

- массовое переименование `Ant`, `Colony`, `queen`, `pheromone` и других legacy-имён;
- шаман, маски, редкие ресурсы и новые большие производственные цепочки;
- сложный AI отдельного жителя;
- полная переделка складской экономики;
- 2 на 2, 3 на 1 и другие режимы до рабочего общего кооператива;
- большой графический полиш до готовой фазы угроз;
- оптимизация под 1000 жителей: текущий игровой лимит намного ниже.

## Общий Definition of Done улучшенной версии

- новая партия начинается без ручной подготовки;
- игрок добывает ресурсы, строит мастерскую и оборону;
- предупреждение даёт время подготовиться;
- тревога заметно меняет поведение поселения;
- неподготовленное поселение страдает, подготовленное выживает;
- после атаки жители чинят и восстанавливают поселение;
- партия заканчивается победой или поражением;
- итог объясняет, что произошло;
- одинаковый seed воспроизводит одинаковую симуляцию;
- typecheck и сценарные проверки зелёные;
- документация обновлена в том же коммите, что код.
