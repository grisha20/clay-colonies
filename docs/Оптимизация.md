## Пункт 3. Не сохранять память племени в обычный snapshot.json - **[Выполнено]**

Что исправить:

Зачем:

Память племени - временное runtime-состояние. Их не обязательно хранить между перезапусками. На текущем сохранении они занимают примерно 747 KB из 1.6 MB.

Файлы:

- `server/src/state/snapshot.ts`

Что сделано:

`saveWorldSnapshot(world)` теперь пишет:

```ts

```

То есть сохранение идет без памяти племени.

Проверка:

- **[Выполнено]** `npm.cmd run check`

Риск:

Низкий. После перезапуска племени потеряют старые тропы памяти племени, но сама симуляция, человечки, королевская пара, комнаты, еда, паук и геномы сохраняются.

Зачем:


## Пункт 10. Разделить durable snapshot и network snapshot - **[Выполнено]**

Что исправить:

Сейчас один `toSnapshot` используется и для сохранения, и для отправки клиенту.

Зачем:

Для сохранения нужны долгоживущие данные. Для клиента нужны видимые данные прямо сейчас. Это разные задачи.

Файлы:

- `server/src/sim/world.ts`
- `server/src/state/snapshot.ts`
- `server/src/net/socket.ts`
- `shared/types.ts`

Как должно быть:

Durable snapshot:

- племени;
- королевской пары;
- молодняк;
- комнаты;
- еда;
- паук;
- геномы/ссылки на геномы;
- минимум runtime-мусора.

Network snapshot:

- видимые человечки;
- видимая еда;
- нужный underground;
- HUD;

- Несколько shard-ов можно запускать в worker threads или отдельных процессах.
- Переход человечков/племён между аренами — редкое событие-«посылка» (упаковать состояние → передать → распаковать), а не бесшовный поток через границу; нет места → не принимать.
- Оффлайн-племена не считаются тик за тиком: при входе игрока делается аналитический fast-forward по прошедшему времени, с отчётом о добыче, потерях, росте молодняка и событиях.
- Походы, отряды и рейды между аренами считаются как timed event + автобой/формулы, а не как честное движение через огромную realtime-карту.
- **Предусловие:** вынести глобальное состояние симуляции (`tickCache`, `antPaths`, паучьи `spiderModes`/`spiderHeadings`/`next*Id`) внутрь `World`, иначе несколько арен в одном процессе будут топтать общее состояние. Делать при работе над аренами, не раньше. Новый продуктовый ориентир — в `docs/Основная мысль Clayfolk.md` и `AGENTS.md`.

Риск:

Высокий. Делать только после стабилизации одиночной арены и сетевого projection.

## Пункт 19B. Offline fast-forward и непрямое управление - **[Не начато]**

Что исправить:

Не держать сотни и тысячи племён в честной realtime-симуляции, когда игроки оффлайн.

Зачем:

Для idle/tamagotchi-игры ощущение живого мира важнее, чем честный CPU-расчёт каждого тика в отсутствие игрока. Если считать всех постоянно, сервер ляжет от количества племён раньше, чем игра станет массовой.

Как должно быть:

- При выходе игрока сохранять timestamp и компактный summary состояния племени/арены.
- Пока игрока нет, активный tick-loop для этой племени не нужен.
- При входе считать `deltaTime` и за короткий аналитический fast-forward получить результат прошедшего времени.
- Отчёт игроку должен быть понятным: добыто еды, вырос молодняк, умерли рабочие, произошли стычки, изменились запасы.
- Формулы fast-forward должны опираться на реальные параметры племени, генома, запасов, угроз и арены, а не выдавать случайную награду из воздуха.
- Непрямое управление задаётся приоритетами: сбор, копание, охрана, уход за молодняком, рейды. Человечки остаются простыми агентами, а игрок управляет племенем через стратегические веса.
- Рейды/отряды/походы между аренами оформлять как событие с таймером: отряд упакован, активная симуляция его не двигает через весь мир, результат считается при завершении.

Риск:

Средний/высокий. Это не локальная CPU-оптимизация, а отдельная модель прогресса. Нужны тесты баланса, чтобы offline fast-forward не ломал экономику и не давал бесконечный рост.
