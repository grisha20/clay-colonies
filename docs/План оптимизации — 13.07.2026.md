# План оптимизации Clayfolk — 13.07.2026

## Цель

Убрать подёргивания на старте и при ускорении симуляции без снижения частоты мира,
числа жителей, качества воды, поведения рыб, точности маршрутов и визуальной плотности.

## Бюджеты

- серверный шаг при обычной стартовой популяции: не более 5 мс в среднем;
- серверный шаг при 160 жителях: не более 20 мс в среднем;
- кадр клиента: не более 16.7 мс для 60 FPS или 33.3 мс при текущем лимите 30 FPS;
- обычный сетевой пакет не должен создавать заметный пик главного потока;
- полный пакет феромонов допускается редко, но не должен провоцировать каскад DOM- и Pixi-перестроений.

## Исходное состояние

После появления озёр безопасный обход вызывал точное вычисление расстояния до всех
сегментов берега при каждой простой проверке воды. `bench:server` не завершался за
180 секунд. После первого исправления весь benchmark выполняется за 5.5 секунды:

- 160 жителей: 14.83 мс/шаг;
- 320 жителей: 19.26 мс/шаг;
- 640 жителей: 47.88 мс/шаг;
- 1000 жителей: 110.55 мс/шаг.

## Этап 1. Геометрия и навигация сервера

Файлы:

- `shared/surfaceTerrain.ts`;
- `server/src/sim/ant/movement.ts`;
- `server/src/sim/building.ts`;
- `server/src/sim/fishing.ts`.

Работы:

1. Простую принадлежность воде проверять полигоном, не полем точной глубины.
2. Точную глубину использовать только для рыб, мелководья и выхода жителя из воды.
3. Кэшировать неизменяемую безопасную сетку озёр для BFS.
4. Динамические стены проверять отдельно, чтобы кэш не устаревал после строительства.
5. Не пересчитывать береговой steering рыб чаще, чем позволяет их скорость.

Риск: расхождение воды на клиенте и сервере. Защита: обе стороны продолжают использовать
общие функции из `shared/surfaceTerrain.ts`.

## Этап 2. Snapshot и сеть

Файлы:

- `server/src/loop.ts`;
- `server/src/net/socket.ts`;
- `server/src/sim/world.ts`;
- `client/src/main.ts`.

Работы:

1. Не сериализовать один вид snapshot повторно для нескольких клиентов.
2. Сохранять wall-clock ограничение отправки при высокой скорости игры.
3. Не создавать временный `Blob` на клиенте только для подсчёта размера JSON.
4. Отделить обработку состояния мира от более редкого обновления DOM-панелей.
5. Перед изменением формата феромонов отдельно измерить JSON parse/unpack; протокол не
   менять без доказанного выигрыша.

Риск: визуально устаревшие показатели. Защита: мир и интерполяция обновляются каждым
пакетом; только текстовые панели ограничены четырьмя обновлениями в секунду.

## Этап 3. Клиентский рендер

Файлы:

- `client/src/render/minimap.ts`;
- `client/src/render/surface/pheromones.ts`;
- `client/src/render/surface/scene.ts`;
- `client/src/render/surface/entities.ts`;
- `client/src/main.ts`.

Работы:

1. Один раз строить статический фон миникарты: землю, озёра и лагеря.
2. Рисовать динамические метки поверх готового слоя.
3. Перестраивать Graphics феромонов только при новой сетке или смене видимых клеток.
4. Сохранять sprite pools; не создавать новые Pixi-объекты в обычном кадре.
5. Следующим отдельным шагом переиспользовать интерполированные объекты жителей, если
   измерения покажут значимую сборку мусора при большой популяции.

Риск: кэш не обновится после новой партии. Защита: ключи включают размеры мира,
координаты входов, ссылки на данные и границы камеры.

## Этап 4. Регрессии и приёмка

1. `npm.cmd run check`.
2. `npm.cmd run bench:server` для 16, 32, 64, 160, 320, 640 и 1000 жителей.
3. `npm.cmd run check:slice`; старый нестабильный тест стены учитывать отдельно.
4. Проверка в игре на 1x, 5x, 20x и Max:
   - камера неподвижна и движется;
   - миникарта открыта и закрыта;
   - видны оба озера и много феромонов;
   - жители обходят воду, рыбы плавают и ловятся;
   - строительство инвалидирует только динамические препятствия.
5. Сравнить HUD `render ms`, интервал пакетов и размер payload до/после.

## Уже реализовано

- быстрая полигональная проверка воды;
- кэш безопасных клеток озёр для поверхностного BFS;
- редкий береговой steering рыб;
- статический слой миникарты;
- кэш Graphics феромонов;
- удаление временного `Blob` на каждый packet;
- ограничение частоты обновления DOM-панелей и миникарты.
- spatial-grid запрос локальной толпы вместо полного прохода по жителям;
- переиспользуемые объекты интерполяции жителей без аллокаций на каждом кадре;
- двойной буфер распакованных феромонов;
- кэш неизменных построек и паутин Pixi;
- ускоренные режимы разбиты на короткие порции максимум по три шага.

## Второй проход

После spatial-grid оптимизации:

- 64 жителя: 3.07 мс/шаг;
- 320 жителей: 11.85 мс/шаг;
- 1000 жителей: 55.10 мс/шаг;
- плотная боевая толпа из 600 жителей: 20.93 мс/шаг в среднем.

Короткие порции ускорения сохраняют целевой темп: `20x` выполняет три шага примерно
раз в 15 мс, `50x` — три шага примерно раз в 6 мс. Это не ускоряет отдельный шаг,
но уменьшает блокировку сетевой отправки и длительность одного event-loop пика.
